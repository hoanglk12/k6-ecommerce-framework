What's Done Well

SharedArray in init context loadCSV/loadJSON wrap data in SharedArray correctly — one copy shared across all VUs, never marshalled through setup().
Module-level GraphQLClient Clients created at module scope (once per VU, before default() runs). TCP connections are reused across iterations.
Exponential backoff with jitter Retry logic in graphql-client.ts uses ±20% jitter to avoid thundering-herd on retries.
Production safety multi-flag ENABLE_PLACE_ORDER + PRODUCTION_CONFIRMED required together before any mutation runs in production.
Request tagging strategy Every request is tagged with site and operation name, enabling fine-grained Grafana filtering.
Clean separation of concerns scenarios/ lib/ config/ tests/ are well separated. Scenarios are pure functions returning ScenarioResult.

Critical Issues

C1 Wrong executor for all tests — ramping-vus when arrival-rate is needed Critical

Every test uses stages, which maps to the ramping-vus executor. With ramping-vus, if the system gets slow, VUs queue waiting for responses — actual throughput (RPS) drops silently. A "200 VU test" might deliver 50 req/min when the system is struggling, giving you no signal about where the breaking point actually is.

For load validation and checkout tests where you want to guarantee X requests-per-minute, the correct executor is ramping-arrival-rate or constant-arrival-rate. These maintain target RPS regardless of response time and are the industry standard for SLA testing.

ramping-vus is still appropriate for realistic concurrent-user simulation (e.g., soak tests simulating N users browsing simultaneously). But it should be a deliberate choice, not the only option.

// Current — throughput varies with system latency
stages: [
  { duration: '2m', target: 50 },
  { duration: '5m', target: 50 },
]

// Better for SLA/load validation — guaranteed throughput
scenarios: {
  pdp_load: {
    executor: 'ramping-arrival-rate',
    startRate: 0,
    timeUnit: '1m',
    preAllocatedVUs: 50,
    maxVUs: 150,
    stages: [
      { duration: '2m', target: 200 },  // 200 req/min
      { duration: '5m', target: 200 },
      { duration: '2m', target: 500 },  // 500 req/min peak
      { duration: '5m', target: 500 },
      { duration: '2m', target: 0   },
    ],
  },
},
C2 Metrics double-counting bug in GraphQL client retry path Critical

In src/lib/graphql-client.ts, when a retryable GraphQL error is encountered, parseResponse() is called inside the retry loop (line 197) — recording graphqlDuration and graphqlRequests — and then called again at line 229 after all retries are exhausted. The same request gets double-counted in every affected metric.

// Inside retry loop — parseResponse() records metrics
const result = this.parseResponse<T>(lastResponse, tags, startTime);
// ... retryable error path continues loop without returning

// After loop — parseResponse() called AGAIN on the same lastResponse
return this.parseResponse<T>(lastResponse, tags, startTime);
// → graphqlDuration + graphqlRequests double-counted

Fix: track whether the final response has already been parsed and skip the second call, or separate metric recording from response parsing into distinct methods.

C3 Product test data hardcoded in test file, bypassing SharedArray Critical

load.test.ts declares product SKUs as a module-level const PRODUCTS object. The entire DataProvider / SharedArray infrastructure in data-provider.ts — and the src/data/products-*.json files — are completely bypassed by the main load test.

This also means four of eight sites still carry PLACEHOLDER-SKU values in production code, guarded only by a runtime Set check that silently no-ops the VU instead of failing fast. All product data should flow through SharedArray to get memory efficiency and consistent data access across VUs.

High-Priority Issues

H1 No scenarios API — only single-journey tests are possible High

The stages shorthand locks every test to a single user journey and a single executor. The scenarios key in k6 options enables multiple concurrent journeys in one test file — each with its own executor, VU pool, start time, and graceful stop duration — which is the correct model for realistic eCommerce load.

A production store doesn't have 200 users all viewing PDPs simultaneously. It has a mix: 70% browsing, 20% searching/filtering, 10% checking out. Without the scenarios API, you cannot model this in a single run and cannot set per-scenario thresholds.

scenarios: {
  pdp_browse: {
    executor: 'ramping-arrival-rate',
    weight: 70,
    stages: [{ duration: '10m', target: 350 }],
  },
  plp_browse: {
    executor: 'constant-arrival-rate',
    rate: 100, timeUnit: '1m',
    preAllocatedVUs: 10, duration: '10m',
  },
  guest_checkout: {
    executor: 'constant-arrival-rate',
    rate: 10, timeUnit: '1m',
    preAllocatedVUs: 5, duration: '10m',
    startTime: '2m', // start after ramp-up
  },
},
H2 No abortOnFail on error-rate thresholds High

All thresholds are defined as strings, which means k6 evaluates them at the end of the test but never aborts early. If the error rate hits 50% at minute 2 of a 16-minute test, k6 keeps running for 14 more minutes hammering a failing system.

// Current — never aborts
'http_req_failed': ['rate<0.01']

// Better — abort early, but allow 30s of ramp-up noise first
'http_req_failed': [{
  threshold: 'rate<0.01',
  abortOnFail: true,
  delayAbortEval: '30s',
}],
H3 check() called inside the library layer High

checkGraphQLResponse() in graphql-client.ts calls check() directly. k6's end-of-test summary groups check results by label — when checks are in a library, the output shows "✓ GraphQL response has no errors" with no indication of which scenario or user journey triggered the failure. Debugging a failing run becomes significantly harder.

check() belongs at the scenario or test layer, where labels can carry meaningful context: "PDP — product detail loads with stock status". Libraries should return structured results and let the caller decide how to assert.

H4 DataProvider.getByVU() breaks under VU ramping High

getByVU() uses exec.vu.idInTest as the data index. With ramping-vus, VU IDs are assigned sequentially as VUs start and are not reassigned when VUs are destroyed between stages. A VU spun up at peak load with ID 100 may map to the same data slot as VU 100 from an earlier stage, causing data collisions and uneven distribution.

// Current — breaks under VU ramping
const index = (exec.vu.idInTest - 1) % this.data.length;

// Better — works correctly across all executors
const index = exec.scenario.iterationInTest % this.data.length;

Medium Issues

M1 No smoke test Medium

The framework has a dry-run (1 VU, skips mutations), but no smoke test that exercises the full happy path at minimal load with real API calls. Best practice is 1–2 VUs, 1–2 iterations, all assertions enabled, no think time — run this in CI before every load test to catch endpoint regressions, config drift, or placeholder SKUs before committing to a 16-minute run.

M2 DataProvider only supports 2 of 8 sites — Fixed Fixed

Resolved. data-provider.ts now exports ProductSite covering all five data keys ('platypus' | 'skechers' | 'drmartens' | 'vans-au' | 'vans-nz') with a SITE_TO_PRODUCT_KEY map for all 8 site IDs. getProductProviderForSite(siteId) accepts any of the 8 site IDs and throws a descriptive error for unknown values.

M3 Remote jslib imports are fragile in CI/CD Medium

Both k6-utils and papaparse are fetched at runtime from jslib.k6.io. k6 caches these locally, but in CI runners with no cache, air-gapped environments, or during CDN outages, tests will fail before they start. These should be vendored into src/lib/vendor/ and imported locally.

// Current — network dependency at runtime
import { randomItem } from 'https://jslib.k6.io/k6-utils/1.4.0/index.js';

// Better — vendored local copy
import { randomItem } from '../vendor/k6-utils.js';
M4 Logger ignores DEBUG env var inside GraphQLClient Medium

The GraphQLClient constructor instantiates its logger with a hardcoded level: 'info', bypassing the environment config's debug flag. Setting DEBUG=true in the environment has no effect on client-level logging. The constructor should use createLogger('GraphQLClient'), which picks up the environment config.

M5 DataProvider instantiated per-call instead of per-VU Medium

Functions like getUserProvider() return new DataProvider(...) on every call. If called inside default() (the per-iteration function), a fresh provider with currentIndex = 0 is created each iteration — making the 'sequential' strategy always return the first item. These provider factory calls must happen at module scope (init context), not inside iteration functions.

Low / Future-Proofing

L1 k6 v0.57+ has native TypeScript support Low

The Webpack + Babel pipeline works today, but as of k6 v0.57 TypeScript is supported natively without a bundler. The main remaining reason to keep Webpack is CopyWebpackPlugin (copying data/ to dist/) and multi-entry bundling. Worth tracking — simplifying the build reduces a meaningful maintenance surface and removes the Babel/webpack version-pinning overhead.

L2 unique data strategy is O(n²) worst-case Low

DataProvider.getUniqueItem() uses rejection sampling to find an unused index. With a small dataset and many VUs exhausting items quickly, this loop can spin for many iterations. A Fisher-Yates pre-shuffle with a sequential pointer is O(1) per access and eliminates the worst case entirely.

L3 No output format standardization for CI/CD Low

No npm scripts emit structured output files. Adding --out json=results/{SITE}-{timestamp}.json to key scripts and committing a baseline result to version control enables trend-over-time tracking without requiring Grafana Cloud, making regressions visible in PR reviews.
