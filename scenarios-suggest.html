Suggested Scenarios

P1 — Highest Authenticated Customer Checkout auth-checkout.test.ts

Why this matters: Registered-customer checkout is a completely different Magento code path from guest checkout. It requires:

  • A customer token (from generateCustomerToken mutation) — adds server-side session + cache pressure
  • A customer cart (bearer-token-scoped), not a guest masked-ID cart
  • Access to saved addresses, loyalty/store-credit, saved payment methods

Guest checkout currently passes does not guarantee registered-customer checkout will hold under load — they hit different Magento code paths entirely.

src/tests/auth-checkout.test.ts — ramping-arrival-rate, 5 orders/min steady, chains loginScenario() → checkout with bearer-token header injection. Reads customers via getUserProvider().

P2 — High Search / SRP (Search Results Page) search-load.test.ts

Why this matters: Search is the highest-traffic read path on footwear/apparel sites during sales and promotions:

  • products(search: "...") is routed through Elasticsearch/OpenSearch, completely bypassing Varnish full-page cache — a cache miss here is unavoidable
  • A spike on "sale" or "white sneakers" during a promotion can collapse the search tier while PDP and PLP remain healthy (false-green on current dashboards)
  • SCENARIOS.md already identifies search as a key customer entry point — it just has no test coverage yet

What to build

  • src/scenarios/search.ts — wraps products(search: "...", pageSize: N), records search_query_time (Trend) and search_results_returned (Rate)
  • src/data/search-terms.json — curated realistic terms per brand (e.g. ["boots", "sale", "white sneakers", "size 10"])
  • src/tests/search-load.test.ts — ramping-arrival-rate, high throughput matching PDP/PLP profile (200 → 500 req/min)
P2 — High Realistic Mixed User Journey journey-load.test.ts

Why this matters: All current tests are single-scenario silos. Real traffic is always mixed concurrently, which produces different behaviour:

  • PDP + PLP together stress Varnish/Redis with competing cache key sets
  • Simultaneous browse and checkout stresses both read replicas and the write DB at the same time
  • Cache-stampede effects only appear under blended load, not isolated load

SCENARIOS.md already documents the recommended traffic split: 62.5% browse (PDP/PLP) · 25% cart/add · 12.5% checkout. A test file implementing this already serves as the most realistic regression check before a production release.

What to build

src/tests/journey-load.test.ts — uses k6's multi-scenario options.scenarios to run three named scenarios concurrently in the documented proportions. Zero new scenario code needed — reuses existing pdpScenario(), plpScenario(), and placeOrderGuestScenario().

scenarios: {
  browse_pdp:     { executor: 'ramping-arrival-rate', ... },  // 62.5%
  browse_plp:     { executor: 'ramping-arrival-rate', ... },  //  25%
  guest_checkout: { executor: 'ramping-arrival-rate', ... },  // 12.5%
}
P3 — Medium Cart / Add-to-Cart Abandonment cart-load.test.ts

Why this matters: Cart operations are a common Magento bottleneck that is hidden by the current test suite:

  • Every cart mutation acquires a row-level DB lock on Magento's quote table — high add-to-cart concurrency causes lock contention even when checkout completion rate is low
  • Over 70% of real carts are never completed — so this load exists in production independently of checkout load
  • Currently, add-to-cart only appears as steps 1–3 of place-order.test.ts, which runs at just 5 orders/min — far below realistic abandonment volume

Metrics scenario_add_to_cart_*, business_cart_value, and business_cart_abandonment are already registered but never recorded.

What to build

src/tests/cart-load.test.ts — extracts and invokes only steps 1–3 of the place-order flow (create guest cart → add item → query cart total). No order placement. Higher throughput than place-order.test.ts to model realistic abandonment volume.

P4 — Lower / Smoke Homepage / CMS cms-load.test.ts

Why this matters: The homepage is the entry point for all direct and campaign traffic and exercises a different infrastructure layer:

  • storeConfig query (config cache) + cmsPage(identifier: "home") (Varnish/FPC) + navigation menu (category-tree cache)
  • These are the fastest-to-serve requests but also the first to fail during a thundering-herd cache miss (e.g. after a cache flush before a promotion)
  • Useful as a lightweight smoke / infrastructure baseline before running heavier scenarios

What to build

  • src/scenarios/cms.ts — storeConfig + cmsPage + category-tree queries
  • src/tests/cms-load.test.ts — high throughput (these should serve <50 ms from cache); useful as smoke baseline
